# SkillGate — AIに「説明書を読むかどうか」を判断させない仕組み

> AIが、渡した説明書を読み飛ばす割合：最大 **46.7%** → SkillGateを通すと **0%**
> （家庭用PCで動く小型AIでの実測。条件と数字の一覧は下の「実測の数字」へ）

![比較デモ：素のAI vs SkillGate](demo.gif)

## その「読まないAI」、こういうやつです

新しい仕事を頼むとき、手順書も一緒に渡しますよね。経費のチェックならチェックの手順書、議事録なら議事録の書き方。ところがAIは、手順書がそこにあるのに開かず、自己流で仕事を始めることがあります。出てきた結果は一見それらしい。でも指定した形式は無視され、判定の基準も手順書と食い違っている。渡した説明書はどこへ行ったのか。

会話が長くなると忘れるんだろう、と最初は思っていました。測ってみると、犯人は別でした。**手順書の冊数**です。5冊なら真面目に開くのに、20冊渡すと「これは知ってる仕事だ」と思い込んで、最大で半分近く（46.7%）を開かずに済ませてしまう。人間の新人と同じで、マニュアルが分厚い棚になった瞬間、読まなくなるんです。

しかも壊れ方は一方向でした。「開かなくていいものを開く」間違いはずっと少ない。壊れるのはいつも「開かない」側です。

だったら、開くかどうかをAIに聞かなければいい。

## SkillGateがやること

玄関に受付係を置きます。あなたの依頼文はAIに届く前に受付係（単語照合のプログラム＋確認役の小さなAI）を通り、受付係が「この仕事にはこの手順書」と決めて、**AIの机の上に手順書を開いて置いてしまう**。AI本人に「読むかどうか」を選ばせません。選ばせないので、サボりようがない。

受付係のルールは「迷ったら置く」です。置きすぎても、手順書には「関係しない場合は無視してよい」と一言添えてあるので実害が出にくい。逆の間違い——必要な手順書の置き忘れ——だけは起きない向きに全体を倒してあります。

仕事が終わったら出口で検品もします。出てきた結果が手順書どおりの形かを機械的に照合して、違っていれば一度だけやり直しをさせる。ここは「見つけて直させる」までで、根絶ではありません（後述）。

「ちゃんと読んで」とお願い文を強くする方向は、試したうえで捨てました。お願い自体が読み飛ばされるからです。お願いではなく、仕組みで渡す——それがこの道具の全部です。

## 実測の数字（Evidence）

エンジニア向けの数字はこの表に集約します。

| 何を測ったか | 結果 |
|---|---|
| 手順書の読み飛ばし（サボり） | 素のAIで最大 **46.7%** → SkillGate経由で **0%**（別の新しい60問でも0%を維持） |
| 手順書の中の「正しい章」を開けるか | 43% → **60%**（改善はしたが未解決） |
| 読んだのに従わない | **6.7〜26.7%** を機械検出・やり直しで約半分を矯正 |
| ルールの書き方だけ変えると | 禁止形 **91%**・手順形 **92%** ＞ ふつうの宣言文 **78%**（理由を書き添えても効かない） |

測定条件：家庭用PCで動く小型AI（Ollama上のqwen3:8b/14b）・日本語の手順書コーパス20本・原則1回測定（n=1）。合否はAIの自己申告ではなく、動作記録の機械照合で判定しています。集計表は [results/benchmark_matrix.md](results/benchmark_matrix.md)、ルールの書き方の実験は [results/文型実験_結果.md](results/文型実験_結果.md)。

## 試してみる（Quick Start）

Python 3.8+のみ。外部ライブラリ依存なし（受付係の確認役とベンチマークにはOllamaが必要）。

**1. 索引を作って、受付係に1問聞いてみる**

```bash
python bench/build_index.py          # corpus/ 配下の手順書から索引を生成
python -c "
from router.route import load_router, route, build_injection
index, trig = load_router()
d = route('経費精算のチェックをお願い', index['skills'], trig)
print(d['skills'])          # → ['keihi-seisan']
print(build_injection(d)[:200])
"
```

Ollamaと`qwen3:8b`があれば確認役のAI込みで動き、なければ単語照合だけの縮退運転になります（止まりはしません）。

**2. 自分の手順書で「読み飛ばし率」を測る**

```bash
python bench/run_bench.py --skills 20 --turn 30 --model qwen3:8b
```

**3. Claude Codeに常駐させる**

送信のたびに受付係が走り、該当する手順書が自動で机に載る使い方です。手順は [router/README_フック導入.md](router/README_フック導入.md)（2ステップ。Ollamaが落ちていても会話は止まらない設計です）。

## 仕組み（How it works）

```
依頼文 ──→ ①単語照合（プログラム・決定論）──┐
       └─→ ②確認役の小さなAIが分類 ─────────┤ ②が主役・①は②が故障した時の予備
                                            ▼
              「読むべき手順書＋開くべき章」を決定
                                            ▼
          手順書本文をプロンプトに強制添付（「関係なければ無視してよい」の一言つき）
                                            ▼
                    AI本体が仕事をする
                                            ▼
          出口：宣言したルールと機械照合 → 違反があれば一度だけやり直し
```

受付係は会話の履歴を見ません。だから「会話が長くなると薄れる」という壊れ方が、そもそも起きない構造です。

なぜAIは説明書を読まないのか——本当の答えは、まだ誰も持っていません。SkillGateが変えたのはその手前です。「読まない」という故障の置き場所を、AIの気まぐれの中から、人間が測って直せる場所へ動かした。それだけのことが、実測では46.7%と0%の差になりました。

## できないこと（Limitations）

正直に書きます。

- 手順書の中の**正しい章を開く精度は6割どまり**。未解決です
- **「読んだのに従わない」は根絶できません**。機械的に見つけて一度やり直させる、まで——やり直しで直るのは約半分です
- 不要な手順書を机に置いてしまうことはゼロではありません（「素のAIの2倍以内」という合格基準の内側。残る誤りの多くは、素のAIでも間違える曖昧な依頼です）
- 測定は小型AI・日本語・原則1回ずつ。大きなモデル・他言語・手順書100本規模では未検証です
- 差し込み口は今のところClaude Codeのフックだけです（フック機構のないチャットアプリでは使えません）
